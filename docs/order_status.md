# Статусы заказа и сценарии работы между сервисами

## 1. Статусы заказа

### `new`
Заказ только что создан в системе, но ещё не отправлен на проверку наличия.

### `waiting_for_stock`
Заказ отправлен на проверку в `catalog_app`, система ждёт ответа по остаткам.

### `confirmed`
Товары есть, резерв успешно выполнен, заказ подтверждён.

### `rejected`
Заказ отклонён, потому что товара нет или его недостаточно.

### `cancelled`
Заказ отменён пользователем или менеджером в допустимый момент.

---

## 2. Допустимые переходы между статусами

Разрешённые переходы:
- `new` → `waiting_for_stock`
- `waiting_for_stock` → `confirmed`
- `waiting_for_stock` → `rejected`
- `new` → `cancelled`
- `waiting_for_stock` → `cancelled`

Запрещённые переходы:
- `confirmed` → `new`
- `confirmed` → `waiting_for_stock`
- `rejected` → `confirmed`
- `cancelled` → `confirmed`

---

## 3. Сценарии работы между сервисами

### Сценарий 1. Успешное создание заказа
1. Пользователь отправляет запрос в `order_app`.
2. `order_app` создаёт заказ со статусом `new`.
3. `order_app` меняет статус на `waiting_for_stock`.
4. `order_app` отправляет событие в RabbitMQ с данными заказа.
5. `catalog_app` получает событие.
6. `catalog_app` проверяет остатки по товарам.
7. Если товаров хватает, `catalog_app` резервирует нужное количество.
8. `catalog_app` отправляет ответное событие об успешном резерве.
9. `order_app` получает ответ и меняет статус заказа на `confirmed`.

### Сценарий 2. Недостаточно товара
1. Пользователь создаёт заказ.
2. `order_app` создаёт запись и отправляет событие на проверку.
3. `catalog_app` получает событие и проверяет остатки.
4. Выясняется, что товара недостаточно.
5. `catalog_app` не резервирует товар.
6. `catalog_app` отправляет событие об отказе.
7. `order_app` получает ответ и меняет статус заказа на `rejected`.

### Сценарий 3. Отмена заказа пользователем
1. Пользователь отправляет запрос на отмену.
2. `order_app` проверяет:
   - принадлежит ли заказ пользователю;
   - можно ли отменить заказ в текущем статусе.
3. Если отмена допустима, заказ получает статус `cancelled`.
4. `order_app` отправляет событие в RabbitMQ.
5. `catalog_app` получает событие и убирает резерв.

### Сценарий 4. Просмотр заказов
- обычный `user` видит только свои заказы;
- `manager` и `admin` видят все заказы.

---

## 4. Дополнительные правила

1. Заказ не должен создаваться пустым.
2. Количество товара в позиции заказа должно быть больше нуля.
3. Пользователь не может смотреть чужой заказ без специальных прав.
4. Пользователь не может отменять чужой заказ.
5. Финальные статусы:
   - `confirmed`
   - `rejected`
   - `cancelled`
